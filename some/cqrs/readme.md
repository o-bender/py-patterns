#CQRS
По большому счёту большинство людей следует этому паттерну неосознанно.
Идея в том что бы разделить взаимодействие с бизнес сущностью на два интерфейса:
- Query - readonly доступ к объекту
- Command - writeonly доступ

Это означает, что Query не должен изменять данные, а Command не должен возвращать
результаты select-ов к БД. По аналогии с REST, где:
- GET запросы только читают,
- POST записывают.

Если взять CQRS относительно БД, то:
  - Query - это select
  - Command - это update и insert. Но Command может возвращать id вставленной
    записи или guid операции если используется Event Sourcing.
